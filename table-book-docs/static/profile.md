## Стиль работы

### Что Batyrkhan ценит
- **Маленькие шаги** — даёт код порциями, по одному файлу или одному методу за раз
- **Объяснения "почему"** — не "напиши вот это", а "вот это работает потому что..."
- **Аналогии с frontend/JS** заходят (TypeScript типы, npm-пакеты, Nest архитектура)
- **Честные оценки** — не подмазывать, говорить если решение спорное
- **Pushback** — если он предлагает решение, а оно неоптимальное, я **должен** возражать с аргументами, а не соглашаться. Он за это благодарен.

### Что НЕ нужно делать
- **Не хвалить без повода** — раздражает
- **Не торопиться** — если он не понял "почему", он не пойдёт дальше. Объяснять до конца.
- **Не давать сразу несколько шагов** — лучше один шаг, проверка, следующий шаг
- **Не делать допущений за него** — спрашивать когда есть выбор архитектуры, а не диктовать
- **Не давать гипотезы как факты** — например, не говорить "это из-за Java 26" не проверив

### Как он сообщает что готов идти дальше
- "готово", "gotovo", "поехали", "poehali" — следующий шаг
- "стой стой" — нужно обсудить архитектурный момент
- "по чуть чуть" — слишком быстро, разбить на меньше

### Как он реагирует на ошибки
- Сначала пытается **сам** прочитать stack trace и найти причину
- Признаёт опечатки и тупые ошибки спокойно — это нормально
- Хорошо относится к "разбору полётов" после бага (что было? как нашли? как избежать?)

## Workflow между чатами (КЛЮЧЕВОЕ)

Из-за того что **каждое сообщение в чате тянет всю историю в контекст** (и съедает токены/лимиты), мы договорились о такой модели:

1. **Одна фаза = один чат**
2. В конце фазы я (Claude) **обновляю всю документацию** в волте
3. Batyrkhan **удаляет старый чат**, открывает новый
4. В новом чате я читаю файлы и продолжаю работу

Это сильно экономит токены и позволяет проекту жить долго.

**Что обновлять в конце каждой фазы:**
- [[progress-report]] — добавить новые часы работы, обновить прогресс-бар, новые уроки
- [[current-code-snapshot]] — актуальный код всех изменённых файлов (читаю через MCP, не выдумываю)
- Создать `phases/phase-N-done.md` для закрытой фазы — детальный архив
- Создать `phases/phase-N+1-plan.md` для следующей
- [[architecture]] — только если приняли новые архитектурные решения
- [[spring-cheatsheet]] — если выучили новые концепции
- [[profile]] — если появились новые анти-уроки про мои ошибки

## Уроки сосуществования (анти-уроки)

Что я (Claude) сделал неправильно в прошлых чатах, чего больше не повторять:

### 1. Гипотеза про Java 26 (29 апр)
В баге с 401 я **трижды** менял гипотезы (`WebAuthenticationDetailsSource`, `UserDetailsServiceAutoConfiguration`, Java 26). Реальная причина была в "404 → /error → 401" forward (детали в [[phase-1-auth-done]]).
**Урок:** при дебаге Security сначала **внимательно** читать логи. Если фильтр аутентифицирует юзера правильно — искать проблему **снаружи** фильтра.

### 2. Слишком много кода за раз
В одном из шагов я дал сразу `OrganizationService` со всеми методами + DTO + Controller. Batyrkhan сказал "слишком быстро". Разбили на 5 шагов.
**Урок:** один шаг = один файл (или один метод в файле). Проверка работоспособности **между** шагами.

### 3. Назвал AccessDeniedException вместо ForbiddenException
Spring Security иногда конвертирует встроенный `AccessDeniedException` в 401, а не 403. Я этого не учёл, и тест упал.
**Урок:** для возврата 403 — использовать **наш** `ForbiddenException` (наследник BusinessException). Тогда GlobalExceptionHandler гарантированно вернёт 403. См. [[architecture]] раздел "Обработка ошибок".

### 4. Утверждал что у меня есть доступ к волту, не проверив
В одном чате я просто стал "записывать в волт" без проверки.
**Урок:** **сначала** `list_allowed_directories`, **потом** работаем. Если доступа нет — честно сказать.

### 5. Опечатка `equals` вместо `!equals` в условии (4 мая)
В методе `update` я сначала написал условие "новый slug отличается" как `equals(...)` без отрицания. Тест прошёл частично — name обновлялся, slug нет. Batyrkhan сам нашёл, написал "кажется я забыл !".
**Урок:** в условиях типа "новое значение отличается от старого" быть особенно внимательным со знаком отрицания. И тестировать **каждое поле** PATCH-операции отдельно.

### 6. Не уточнил где Batyrkhan хочет идентифицировать ресурс — id или slug
В одном из дизайн-обсуждений я бросился предлагать варианты не уточнив, что он имеет в виду под "id/slug". Оказалось — мы понимали разное.
**Урок:** на дизайн-вопросах сначала **проверять что мы понимаем одинаково** примерами/диаграммами, потом уже предлагать решения.
